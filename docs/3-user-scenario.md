# TodoListApp 사용자 시나리오 문서

**문서 버전**: 1.0  
**작성일**: 2026-05-13  
**참조 문서**: [도메인 정의서](./1-domain-definition.md), [PRD](./2-prd.md)  
**변경 이력**: 최초 작성 → 실제 구현 기준 보정 (2026-05-14): SC-001 회원가입 후 토큰 즉시 발급·/todos 리다이렉트, SC-003/SC-006 request body camelCase, SC-007 카테고리 삭제 응답 204 No Content

---

## 1. 문서 개요

### 1.1 목적
이 문서는 TodoListApp의 핵심 기능을 사용자 페르소나별로 구체적인 업무 상황에서 어떻게 사용할지를 시나리오 형태로 정의한다. 개발팀이 사용자 관점에서 필요한 기능을 명확히 이해하고, QA 팀이 테스트 케이스를 설계하는 데 참고할 수 있다.

### 1.2 시나리오 작성 기준
- **액터(Actor)**: 페르소나 A(김지수), 페르소나 B(박서연), 페르소나 C(이민호)
- **사전 조건**: 시나리오 시작 직전의 상태
- **트리거**: 사용자 행동의 시작점
- **기본 흐름**: 정상 경로로 진행되는 주요 단계들 (번호 매김)
- **대안 흐름**: 예외 상황이나 분기 경로 (if-then 형식)
- **사후 조건**: 시나리오 완료 후의 시스템 상태

## 2. 사용자 시나리오 상세 정의

### SC-001: 신규 사용자 회원가입

**시나리오 ID**: SC-001  
**액터**: 페르소나 B — 박서연 (26세, 스타트업 마케터)  
**우선순위**: P0 (핵심 기능)  
**복잡도**: 낮음

#### 2.1.1 사전 조건
- 사용자가 TodoListApp 로그인 페이지(`/login`)에 접근함
- 아직 계정을 가지고 있지 않음
- 회원가입 링크(`/register`)가 보임

#### 2.1.2 트리거
사용자가 회원가입 링크를 클릭하여 회원가입 폼 페이지로 이동

#### 2.1.3 기본 흐름
1. 사용자가 회원가입 폼을 본다 (이름, 이메일, 비밀번호, 비밀번호 확인 입력 필드)
2. 사용자가 이름 입력란에 "박서연"을 입력한다
3. 사용자가 이메일 입력란에 "seoyeon.park@startup.co.kr"을 입력한다
4. 사용자가 비밀번호 입력란에 "Marketing2024"를 입력한다 (8자 이상, 영문+숫자)
5. 사용자가 비밀번호 확인란에 동일한 비밀번호를 재입력한다
6. 사용자가 "회원가입" 버튼을 클릭한다
7. 시스템이 입력 데이터를 검증한다:
   - 이메일 형식이 RFC 5322를 준수하는지 확인
   - 비밀번호가 최소 8자 이상인지 확인
   - 비밀번호에 영문과 숫자가 모두 포함되어 있는지 확인
   - 두 비밀번호가 일치하는지 확인
   - 해당 이메일이 이미 가입되어 있지는 않은지 확인
8. 시스템이 회원가입을 처리한다:
   - 새 사용자 레코드를 DB에 생성 (이메일, bcrypt 해시 비밀번호, 이름)
   - 사용자에게 기본 카테고리 3개를 자동 생성: "일반"(#6B7280), "업무"(#3B82F6), "개인"(#10B981)
   - Access Token (1시간 만료) + Refresh Token (7일 만료)을 즉시 발급하고 DB에 Refresh Token 저장
   - 응답: `{ user, accessToken, refreshToken }`
9. 클라이언트가 토큰을 Zustand 메모리 스토어에 저장하고 할일 목록 페이지(`/todos`)로 자동 리다이렉트한다 (별도 로그인 불필요)

#### 2.1.4 대안 흐름

**AF-001: 유효하지 않은 이메일 형식**
- Step 7에서 이메일 형식 검증 실패
- 시스템이 사용자에게 에러 메시지: "유효한 이메일 형식을 입력해주세요"
- 사용자가 다시 이메일을 수정하고 회원가입을 재시도

**AF-002: 이미 사용 중인 이메일**
- Step 7에서 이메일 중복 확인 시 기존 가입 사용자 발견 (에러 코드: USER_001)
- 시스템이 사용자에게 에러 메시지: "이미 가입되어 있는 이메일입니다"
- 사용자가 다른 이메일을 입력하거나 로그인 페이지로 이동

**AF-003: 약한 비밀번호**
- Step 7에서 비밀번호 정책 검증 실패
- 시스템이 사용자에게 에러 메시지: "비밀번호는 최소 8자 이상이며 영문과 숫자를 모두 포함해야 합니다"
- 사용자가 비밀번호를 수정하여 재시도

**AF-004: 비밀번호 불일치**
- Step 7에서 두 비밀번호 입력값이 다름을 확인
- 시스템이 사용자에게 에러 메시지: "비밀번호가 일치하지 않습니다"
- 사용자가 비밀번호 확인 필드를 다시 입력

#### 2.1.5 사후 조건
- 사용자 테이블에 새로운 사용자 레코드 생성
- 해당 사용자의 카테고리 테이블에 "일반"(#6B7280), "업무"(#3B82F6), "개인"(#10B981) 3개 기본 카테고리 자동 생성
- Access Token + Refresh Token이 발급되어 클라이언트 Zustand 메모리 스토어에 저장됨
- 사용자가 할일 목록 페이지(`/todos`)로 이동하여 즉시 서비스 이용 가능

### SC-002: 로그인 및 오늘 할일 확인

**시나리오 ID**: SC-002  
**액터**: 페르소나 A — 김지수 (38세, IT 팀장)  
**기기**: 모바일 웹 (스마트폰)  
**우선순위**: P0 (핵심 기능)  
**복잡도**: 낮음

#### 2.2.1 사전 조건
- 사용자가 이미 회원가입을 완료했음
- 사용자가 모바일 기기에서 TodoListApp 로그인 페이지(`/login`)를 방문함
- 로그인하지 않은 상태

#### 2.2.2 트리거
사용자가 아침 출근 중 모바일 웹에서 오늘의 할일을 확인하고 싶어서 로그인함

#### 2.2.3 기본 흐름
1. 사용자가 로그인 페이지의 이메일 입력란에 "jisoo.kim@techcorp.kr"을 입력한다
2. 사용자가 비밀번호 입력란에 "DevTeamLead2024"를 입력한다
3. 사용자가 "로그인" 버튼을 클릭한다
4. 시스템이 입력 데이터를 검증한다:
   - 이메일 형식 확인
   - 해당 이메일의 사용자를 DB에서 조회
   - 입력한 비밀번호와 저장된 bcrypt 해시값을 비교
5. 로그인 성공:
   - 시스템이 JWT Access Token (1시간 만료)을 발급
   - 시스템이 JWT Refresh Token (7일 만료)을 발급하고 DB에 저장
   - 클라이언트의 Zustand 메모리 스토어에 두 토큰을 저장 (localStorage 미사용)
6. 시스템이 할일 목록 페이지(`/todos`)로 자동 리다이렉트
7. 페이지 로드 시:
   - 클라이언트가 저장된 Access Token을 Authorization 헤더에 포함하여 API 요청
   - 시스템이 `GET /api/v1/todos?period=today&status=all` API 호출
8. 서버가 다음을 실행:
   - Authorization 헤더에서 Token 추출 및 검증
   - JWT payload에서 user_id 추출
   - 해당 사용자의 오늘(2026-05-13) 할일 목록을 조회 (완료/미완료 모두)
   - 종료 예정일 오름차순, 생성일 내림차순으로 정렬
9. 시스템이 할일 목록을 모바일 UI에 표시한다:
   - 할일 제목, 카테고리, 종료일 정보를 카드 형식으로 표시
   - 완료 여부를 체크박스로 표시
   - 총 할일 개수, 완료된 개수, 미완료 개수 요약 표시
   - 필터 바 (카테고리, 기간, 완료 여부) 제공
10. 사용자가 할일 목록을 모바일 화면에서 확인한다

#### 2.2.4 대안 흐름

**AF-001: 이메일 또는 비밀번호 오류**
- Step 4에서 이메일을 찾지 못하거나 비밀번호가 일치하지 않음 (에러 코드: USER_002)
- 시스템이 사용자에게 에러 메시지: "이메일 또는 비밀번호가 올바르지 않습니다"
- 사용자가 입력값을 다시 확인하고 로그인을 재시도

**AF-002: Authorization 헤더 누락**
- Step 7의 API 요청이 Authorization 헤더 없이 전송됨
- 시스템이 401 Unauthorized 응답 (에러 코드: AUTH_001)
- 클라이언트가 사용자를 로그인 페이지로 리다이렉트

**AF-003: Access Token 만료**
- 사용자가 로그인 후 1시간 이상 경과하여 Step 7의 API 요청 시 Token이 만료됨 (에러 코드: AUTH_002)
- 시스템이 401 Unauthorized 응답
- 클라이언트가 Refresh Token을 사용하여 자동으로 새 Access Token을 재발급받음 (`POST /api/v1/auth/refresh`)
- 새로운 Access Token으로 API 요청을 재시도
- (토큰 재발급 로직은 SC-010 참조)

**AF-004: 모바일 네트워크 지연**
- Step 8의 API 응답이 지연되는 경우
- 클라이언트에서 로딩 스피너를 표시
- 5초 이상 응답 없을 시 타임아웃 에러 메시지 표시
- 사용자가 재시도 버튼을 클릭

#### 2.2.5 사후 조건
- 사용자가 정상적으로 로그인됨 (Access Token 발급)
- 할일 목록 페이지가 로드되고 오늘 할일 목록이 표시됨
- 모바일 화면에 최적화된 UI로 할일 정보가 표시됨

### SC-003: 새 할일 등록

**시나리오 ID**: SC-003  
**액터**: 페르소나 C — 이민호 (44세, 프리랜서 디자이너)  
**기기**: PC (데스크탑)  
**우선순위**: P0 (핵심 기능)  
**복잡도**: 중간

#### 2.3.1 사전 조건
- 사용자가 이미 로그인 완료
- 사용자가 할일 목록 페이지(`/todos`)에 있음
- 사용자가 기존 카테고리: "일반", "업무", "개인", "클라이언트_A", "클라이언트_B" 5개를 보유함

#### 2.3.2 트리거
새로운 클라이언트로부터 프로젝트 의뢰가 들어와 할일 목록에 추가하고 싶음. 페이지의 "새 할일" 버튼을 클릭함.

#### 2.3.3 기본 흐름
1. 사용자가 "새 할일" 버튼을 클릭하여 할일 등록 폼(`/todos/new`)으로 이동
2. 폼 페이지가 로드되고 다음 필드가 표시됨:
   - 제목 (필수, 최대 200자)
   - 설명 (선택, 최대 2000자)
   - 카테고리 (필수, 드롭다운)
   - 종료 예정일 (필수, 날짜 선택기)
3. 사용자가 제목 입력란에 "클라이언트C 웹사이트 리디자인 - UI 프로토타입"을 입력한다
4. 사용자가 설명 입력란에 다음을 입력한다:
   "클라이언트C의 신규 웹사이트 리디자인 프로젝트. 
   요구사항: 모바일 반응형, 다크 모드 지원, 애니메이션 포함
   회의 날짜: 2026-05-15 10:00 AM
   상세 기획서는 이메일로 전달 예정"
5. 사용자가 카테고리 드롭다운을 열어 "클라이언트_C"를 선택한다
   (만약 카테고리가 없다면, 유저는 여기서 카테고리 추가 기능을 사용할 수 있음 - SC-006 참조)
6. 사용자가 종료 예정일 선택기에서 "2026-05-31"을 선택한다 (약 18일 뒤)
7. 사용자가 "등록" 버튼을 클릭한다
8. 클라이언트가 입력 데이터를 검증한다:
   - 제목이 비어있지 않고 200자 이하인지 확인
   - 카테고리가 선택되었는지 확인
   - 종료 예정일이 선택되었는지 확인
   - 설명이 2000자 이하인지 확인
9. 검증 성공 시, 클라이언트가 `POST /api/v1/todos` API를 호출:
   ```json
   {
     "title": "클라이언트C 웹사이트 리디자인 - UI 프로토타입",
     "description": "클라이언트C의 신규 웹사이트 리디자인 프로젝트...",
     "categoryId": 5,
     "dueDate": "2026-05-31"
   }
   ```
10. 서버가 다음을 실행:
    - Authorization 헤더에서 Token 검증
    - JWT에서 user_id 추출
    - category_id가 해당 사용자 소유 카테고리인지 검증
    - due_date 형식이 DATE 타입인지 검증
    - 새로운 할일 레코드를 DB에 삽입:
      ```
      INSERT INTO todos (user_id, category_id, title, description, due_date, is_completed, created_at, updated_at)
      VALUES (123, 5, '...', '...', '2026-05-31', false, NOW(), NOW())
      ```
11. 서버가 201 Created 응답과 함께 생성된 할일 객체를 반환:
    ```json
    {
      "id": 456,
      "userId": 123,
      "categoryId": 5,
      "title": "클라이언트C 웹사이트 리디자인 - UI 프로토타입",
      "description": "...",
      "dueDate": "2026-05-31",
      "isCompleted": false,
      "createdAt": "2026-05-13T14:30:45.123Z",
      "updatedAt": "2026-05-13T14:30:45.123Z"
    }
    ```
12. 클라이언트가 할일 목록 페이지로 자동 리다이렉트하고 "할일이 등록되었습니다" 성공 메시지를 표시한다
13. 새로 등록된 할일이 목록에 나타난다 (가장 최근 생성 순서로)

#### 2.3.4 대안 흐름

**AF-001: 필수 필드 누락**
- Step 8에서 제목, 카테고리, 또는 종료 예정일 중 하나 이상이 누락됨
- 클라이언트가 해당 필드 옆에 빨간 오류 표시 및 메시지: "필수 항목입니다"
- 사용자가 누락된 필드를 채우고 다시 등록 버튼 클릭

**AF-002: 입력값이 최대 길이 초과**
- Step 8에서 제목이 200자를 초과하거나 설명이 2000자를 초과함
- 클라이언트가 실시간으로 글자 수를 표시하고 초과 시 경고
- 에러 메시지: "제목은 200자 이내, 설명은 2000자 이내여야 합니다"
- 사용자가 텍스트를 수정

**AF-003: 선택한 카테고리가 사용자 소유 아님**
- Step 10에서 서버가 category_id를 검증할 때 다른 사용자의 카테고리임을 발견 (에러 코드: AUTH_004)
- 시스템이 에러 응답 (403 Forbidden): "접근 권한이 없습니다"

   - 불일치하면 401 오류 (USER_002): "비밀번호가 올바르지 않습니다"
   - 일치하면 계속 진행
   - **Step 7.4**: 트랜잭션 시작
   - **Step 7.5**: 해당 사용자의 모든 할일 삭제 (CASCADE)
     ```sql
     DELETE FROM todos WHERE user_id = 123
     ```
     - 영향받은 행: 23개
   - **Step 7.6**: 해당 사용자의 모든 카테고리 삭제
     ```sql
     DELETE FROM categories WHERE user_id = 123
     ```
     - 영향받은 행: 5개
   - **Step 7.7**: 해당 사용자의 모든 Refresh Token 삭제
     ```sql
     DELETE FROM refresh_tokens WHERE user_id = 123
     ```
     - 영향받은 행: 2개
   - **Step 7.8**: 사용자 레코드 삭제
     ```sql
     DELETE FROM users WHERE id = 123
     ```
     - 영향받은 행: 1개
   - **Step 7.9**: 트랜잭션 커밋
8. 서버가 200 OK 또는 204 No Content 응답 반환:
   ```json
   {
     "message": "회원 탈퇴가 처리되었습니다. 모든 데이터가 삭제되었습니다."
   }
   ```
9. 클라이언트가 다음을 실행:
   - Zustand 메모리 스토어에서 Access Token, Refresh Token 초기화
   - 전역 상태(Zustand)에서 인증 상태 초기화
   - 사용자를 로그인 페이지(`/login`)로 리다이렉트
10. 사용자가 로그인 페이지를 보고 회원탈퇴가 완료됨을 확인한다
11. 사용자가 기존 이메일로 재가입을 시도하면 새로운 계정으로 가입 가능함 (기존 데이터 없음)

#### 2.9.4 대안 흐름

**AF-001: 비밀번호 오류**
- Step 7.3에서 입력한 password가 저장된 해시와 불일치
- 서버가 401 Unauthorized 응답 (에러 코드: USER_002): "비밀번호가 올바르지 않습니다"
- 클라이언트가 오류 메시지 표시
- 사용자가 비밀번호를 다시 확인하고 재입력
- 회원탈퇴는 진행되지 않음

**AF-002: 사용자가 확인 모달에서 [취소] 선택**
- Step 4에서 [취소] 버튼 클릭
- API 호출 없음
- 사용자가 설정 페이지로 그대로 유지
- 회원탈퇴 안 됨

**AF-003: 트랜잭션 실패 (DB 오류)**
- Step 7.5~7.9 중 예기치 않은 오류 발생
- 서버가 500 Internal Server Error 응답 (에러 코드: SERVER_001)
- 트랜잭션 롤백되어 모든 데이터 그대로 유지됨
- 사용자 탈퇴 안 됨
- 클라이언트가 사용자에게 오류 메시지 표시
- 사용자가 나중에 재시도 가능 (또는 관리자에 문의)

**AF-004: Authorization Header 누락 또는 Token 만료**
- Step 7.1에서 Token 검증 실패
- 서버가 401 Unauthorized 응답 (에러 코드: AUTH_001 또는 AUTH_002)
- 사용자가 로그인 페이지로 리다이렉트됨

#### 2.9.5 사후 조건
- 사용자 테이블에서 해당 사용자 레코드 즉시 물리 삭제
- 해당 사용자의 모든 할일(23개) 삭제 (CASCADE)
- 해당 사용자의 모든 카테고리(5개) 삭제 (CASCADE)
- 해당 사용자의 모든 Refresh Token(2개) 삭제 (CASCADE)
- 클라이언트의 Zustand 메모리 스토어에서 토큰 정보 초기화
- 사용자가 로그인 페이지로 리다이렉트됨
- 동일 이메일로 새로운 계정 가입 가능함

### SC-010: Access Token 만료 후 자동 재발급

**시나리오 ID**: SC-010  
**액터**: 페르소나 A — 김지수 (38세, IT 팀장)  
**기기**: PC  
**우선순위**: P0 (핵심 기능)  
**복잡도**: 높음 (기술 시나리오)  

#### 2.10.1 사전 조건
- 사용자가 이전에 로그인 완료했음 (지난 1시간 전)
- 사용자가 Access Token을 Zustand 메모리 스토어에 보유하고 있음
- Access Token의 만료 시간: 1시간
- 현재 시간: 로그인 후 1시간 5분
- Access Token은 이미 만료된 상태
- Refresh Token은 여전히 유효 (7일 내)

#### 2.10.2 트리거
사용자가 오래 있던 페이지에서 새로운 API 요청을 시도함 (예: 할일 새로고침). 서버가 Token 만료를 감지함.

#### 2.10.3 기본 흐름
1. 사용자가 할일 목록 페이지에서 새로고침 버튼을 클릭하거나 새 필터를 적용하려고 함
2. 클라이언트가 API 요청을 준비한다:
   ```
   GET /api/v1/todos?category_id=2&status=undone
   Authorization: Bearer {expired_access_token}
   ```
3. 서버가 API 요청을 받고 Token을 검증한다
4. 서버가 Token이 만료되었음을 감지한다 (exp claim 비교)
5. 서버가 401 Unauthorized 응답을 반환한다 (에러 코드: AUTH_002):
   ```json
   {
     "error": {
       "code": "AUTH_002",
       "message": "Access Token이 만료되었습니다"
     }
   }
   ```
6. 클라이언트가 401 응답을 받는다
7. 클라이언트의 API 인터셉터(TanStack Query v5 훅 → api 클라이언트 인터셉터)가 자동으로 다음을 실행한다:
   - 현재 요청을 잠금(queue)
   - Refresh Token이 Zustand 메모리 스토어에 존재하는지 확인
   - Refresh Token이 존재하면, `POST /api/v1/auth/refresh` API를 호출:
     ```json
     {
       "refreshToken": "{saved_refresh_token}"
     }
     ```
   - Refresh Token이 없으면 → Step 12로 이동 (로그인 페이지로 리다이렉트)
8. 서버가 Refresh 요청을 받고 다음을 실행한다:
   - 제공된 refreshToken을 검증한다 (JWT 서명 확인)
   - JWT payload에서 user_id 추출
   - refresh_tokens 테이블에서 해당 토큰 조회:
     ```sql
     SELECT * FROM refresh_tokens WHERE token = '{provided_token}' AND user_id = {user_id}
     ```
   - 토큰이 존재하고 expires_at > NOW() 인지 확인
   - 조건을 만족하면, 새로운 Access Token 발급:
     - 페이로드: {user_id, email, iat, exp: NOW + 1hour}
     - 서명: JWT_SECRET으로 HS256 서명
   - 새로운 Access Token과 함께 200 OK 응답 반환:
     ```json
     {
       "accessToken": "{new_access_token}",
       "expiresIn": 3600
     }
     ```
9. 클라이언트가 새로운 Access Token을 받는다
10. 클라이언트가 Zustand 메모리 스토어의 Access Token을 새로운 값으로 업데이트한다
11. 클라이언트가 잠금된 원래 요청을 새로운 Access Token으로 재시도한다:
    ```
    GET /api/v1/todos?category_id=2&status=undone
    Authorization: Bearer {new_access_token}
    ```
12. 서버가 새로운 Token으로 원래 요청을 처리하고 정상 응답한다:
    ```json
    {
      "data": [ /* 할일 목록 */ ],
      "pagination": { ... }
    }
    ```
13. 클라이언트가 응답을 받고 UI에 데이터를 표시한다
14. 사용자가 새로고친 할일 목록을 본다 (중단 없이 자동으로 처리됨)

#### 2.10.4 대안 흐름

**AF-001: Refresh Token이 만료됨**
- Step 8에서 refresh_tokens 테이블에서 토큰을 찾거나 expires_at > NOW() 조건 불만족
- 서버가 401 Unauthorized 응답 (에러 코드: AUTH_005): "Refresh Token이 만료되었습니다"
- 클라이언트가 Zustand 메모리 스토어에서 모든 토큰 초기화
- 클라이언트가 사용자를 로그인 페이지로 리다이렉트
- 사용자가 다시 로그인해야 함

**AF-002: Refresh Token이 DB에 없음 (회원탈퇴 또는 강제 로그아웃)**
- Step 8에서 refresh_tokens 테이블에 해당 토큰이 없음
- 서버가 401 Unauthorized 응답 (에러 코드: AUTH_005): "유효하지 않은 Refresh Token입니다"
- 클라이언트가 사용자를 로그인 페이지로 리다이렉트

**AF-003: Refresh 요청 중 서버 오류**
- Step 8의 토큰 발급 중 예기치 않은 오류 발생
- 서버가 500 Internal Server Error 응답 (에러 코드: SERVER_001)
- 클라이언트가 재시도 메커니즘 작동 (최대 3회 재시도)
- 계속 실패하면 사용자를 로그인 페이지로 리다이렉트

**AF-004: Refresh Token이 Zustand 스토어에 없음**
- Step 7에서 Zustand 메모리 스토어에 Refresh Token이 없음 (예: 페이지 새로고침으로 인한 메모리 초기화 또는 불완전한 로그인)
- 클라이언트가 사용자를 로그인 페이지로 리다이렉트
- 사용자가 다시 로그인해야 함

**AF-005: 네트워크 오류로 Refresh 요청 실패**
- Step 7의 Refresh API 호출 중 네트워크 오류 발생
- 클라이언트가 재시도 메커니즘 작동 (최대 3회)
- 모두 실패하면 사용자에게 오류 메시지 표시
- 사용자가 수동으로 새로고침하거나 로그인 페이지로 이동

#### 2.10.5 사후 조건
- 새로운 Access Token이 발급되고 클라이언트에 저장됨
- 기존 Access Token은 더 이상 사용되지 않음
- 원래 API 요청이 성공적으로 처리됨
- 사용자는 중단 없는 서비스 경험 (자동 재발급)
- Refresh Token은 변경되지 않음 (그대로 유지, 7일 만료 유효)

## 6. 주요 비즈니스 규칙 체크리스트

각 시나리오 구현 시 다음 비즈니스 규칙이 항상 지켜지는지 확인하자.

- [x] **데이터 격리**: 모든 API는 JWT user_id로 검증하여, 사용자는 자신의 데이터만 접근 가능
- [x] **기본 카테고리 불변성**: "일반", "업무", "개인"은 삭제 불가 (is_default = true)
- [x] **할일 안전 삭제**: 카테고리 삭제 시 할일은 "일반"으로 이전 (손실 없음)
- [x] **비밀번호 보안**: bcrypt (salt rounds: 12) 해싱, 평문 저장 금지
- [x] **Token 만료 처리**: Access Token 1시간, Refresh Token 7일
- [x] **회원 탈퇴 즉시 삭제**: 소프트 삭제 없이 물리 삭제 (CASCADE)
